# 実装指示書：2.5Dサードプレイス化 ＋ ボード機能

この文書は実装担当（Sonnet）への指示書です。`README.md` と `SPEC.md` を先に読んでから着手してください。
判断に迷ったら `SPEC.md` の「2. 設計原則」に立ち返ること。

---

## 0. ゴール（何を作るか）

現状は「平面の見下ろし（DOMベースの2Dフロア）」でテント内を表示している。これを、既に別ページで検証済みの
**2.5Dパース＋ビルボード表示（`client/src/mockup/` の内容）に置き換え**、その世界の中に
**ユーザーが「ボードを立てる」ことができ、そのボードで画面共有・テキストのやりとりができる**機能を追加する。

ユーザーの完成イメージ：
> 仮想空間上のホワイトボードにみんなで集まっている。そこに画面を共有したり、近くの掲示板にテキストを打ち込んだりできる。

---

## 1. 絶対に守ること（既存機能を壊さない）

現在このアプリは本番稼働しており、以下が動いている。**すべて維持すること**：

- LiveKit接続（`App.jsx`）、入室フォーム、接続中/失敗のフィードバック
- 位置・アバター種別の同期（`TentState.jsx`。データチャネル`position`）
- アバター2系統（石🪨・草🌿＝非人間/破線枠・話さない、人🧍＝人型/実線枠・話す可能性）と切替、**非人間はマイク強制ミュート**（`AvatarSprite.jsx` / `CallScreen.jsx`）
- 空間オーディオ（`SpatialAudio.jsx`。距離減衰＋ステレオパン）
- WASDキー移動＋ドラッグ移動＋**ホップ演出**（歩行の代わり。`TentState.jsx`）
- 発話中の視覚表示
- チャット（`ChatState.jsx` ＝常時マウントの状態保持＋データチャネル`chat`、`ChatPanel.jsx` ＝見た目）
- 画面共有（`ScreenShareStage.jsx`）＋共有画面への描き込み（`DrawingOverlay.jsx`。データチャネル`draw`。ペン/丸/消しゴム）
- 退出ボタン、RPG風テーマ（`index.css`）、スマホ対応（横スクロールなし・タップ44px・iOSズーム防止・`100svh`）

### 特に壊しやすい・必ず踏襲する実装上の注意
- **接続完了（`useConnectionState()` が `Connected`）まで一切 `publishData` しない**。接続前に送るとLiveKit内部の送信路が失敗状態でキャッシュされ、以降チャット含む全データ送信が失敗し続ける（`TentState.jsx`の`publish`ガードを踏襲）。ボード同期でも同じガードを必ず入れること。
- `publishData`（データチャネル）は**自分には配信されない**。自分の操作結果は**ローカルに即時反映**してから送る（チャット・描き込みと同じ方式）。
- 位置・アバター種別・ボードなど**その場限りでよいものは全部データチャネル**。**サーバー/DBは使わない**（今回、永続化・ログ・掲示板の“残す”機能はスコープ外。§6参照）。
- テキスト入力欄にフォーカスがある間はWASD移動を無効化する既存判定を維持。
- 各機能は**2タブ（別ブラウザ/シークレット）で相互に見えるか**を必ず確認（ヘッドレスのプレビューでは実WebRTCが張れないため、`npm run build && npx vite preview` か本番URLで確認する。ローカルの実ブラウザ2つでの確認が確実）。

## 2. 設計方針（重要）

**3Dの世界は「居る・歩く・集まる」を担当。実際のコミュニケーションUIは、既に動いている2Dパネルを“ボードをタップしたときのフォーカス表示”として再利用する。**

- 3D内に画面共有UIやチャットUIを作り直さない。ボードは「集まる目印＋入口」。タップ/接近したら**既存の`ScreenShareStage`+`DrawingOverlay` / `ChatPanel`をオーバーレイで開く**。閉じると世界に戻る。
- これにより、動作実績のある通信機能を壊さず、見た目だけ2.5D化できる。

### ボードは2種類に分ける（決定事項）
- **共有ボード（画面共有用）**：囲んで一緒に画面を見る／描き込む
- **テキストボード（チャット用）**：メッセージを読む／打ち込む
- 別オブジェクトにするが、近くにワンセットで置けるようにして「ホワイトボードのある一角」感を出す。

### 今回のスコープでの割り切り
- ボード・画面共有・チャットの中身は当面**ルーム全体で共有**でよい（近接スコープ＝「近くの人にだけ見える」は将来。§6）。
- ボードは「立てる目印」として機能すればよい。まずは**集まる体験**を作る。

## 3. 実装フェーズ（順番に。各ステップで動作確認→lint/build→コミット→デプロイ確認→次へ）

`SPEC.md`同様「動くものを確認してから次へ」。各ステップ完了時に `README.md`・`SPEC.md`（Phase 1の該当や新規節）を更新し、`git commit -m "説明"` → push → GitHub Actions成功 → 本番URL確認、まで行う。

### Step 0：react-three-fiber を Vite開発サーバーで動くようにする（前提・ブロッカー）
現状 `mockup.html` は本番ビルドでは動くが、`npm run dev`（Vite開発サーバー）では r3f 由来の
`Invalid hook call`（Reactの重複インスタンス）が出る。**r3fを本番アプリ本体に組み込むと、開発サーバーでアプリ全体がこのエラーになる**ため、先に解決する。
- 既に `vite.config.js` に `resolve.dedupe: ['react','react-dom']` は入れてある（だけでは直らなかった）。
- 候補：`optimizeDeps.include` に `react`, `react-dom`, `react-dom/client`, `react/jsx-runtime`, `react/jsx-dev-runtime`, `@react-three/fiber`, `@react-three/drei`, `three` をまとめて入れて単一インスタンスに強制する／`npm ls react` で重複がないか確認／`node_modules/.vite` を消して再起動。
- **完了条件**：`npm run dev` でアプリ（および mockup）を開いて `Invalid hook call` が出ず、3Dが表示される。どうしても開発サーバーで解決できない場合は、その旨を記録し「検証は `npm run build && npx vite preview` で行う」運用に切り替えてよい（ただしまず解決を試みること）。

### Step 1：テント内表示を 2.5D(r3f) に置き換える（既存機能は全維持）
`client/src/mockup/Mockup.jsx` を土台に、本番の `TentView.jsx` を「DOM 2Dフロア」→「r3fの2.5Dシーン」へ置き換える。
- **位置・アバター種別の正は引き続き `TentState.jsx`**（ここは触らない。ソース・オブ・トゥルース）。r3fシーンは`useTentState()`から`myPos`/`others`/`avatarType`/`hopping`/`localIdentity`を読み、参加者をビルボードのアバターとして配置するだけにする。
- 移動：既存のWASD＋ドラッグに加え、mockupの**タップ/ドラッグで移動先指定**（床のraycast）も取り込む。`updateMyPos`を呼ぶ形にして同期・ホップ・空間オーディオがそのまま効くようにする。
- カメラは自分を追従する斜め見下ろし（mockup同様）。
- アバターの見た目は mockup の絵文字ビルボード方式を採用しつつ、**石/草/人の区別（枠線）・発話リング・ホップ**を必ず反映（`AvatarSprite`相当の意味を保つ）。3Dなので`AvatarSprite.jsx`はr3f用に作り直す or 併存させる。
- 座標系の対応に注意：`TentState`は0〜1正規化(x,y)。3Dワールドの床(x,z)へ線形変換する薄い関数を用意（例：`worldX = (px-0.5)*FIELD`, `worldZ = (py-0.5)*FIELD`）。空間オーディオの距離計算は正規化座標のままでよい（`SpatialAudio.jsx`は変更不要のはず）。
- **完了条件**：入室すると2.5Dのテントが表示され、これまで通り「石/人の切替・マイクミュート・移動（WASD/ドラッグ/タップ）・ホップ・空間オーディオ・発話表示・チャット・画面共有・退出」が全部動く。スマホで横スクロールなし。2タブで相手のアバターと移動が見える。

### Step 2：「ボードを立てる」
- 操作バーに「🪧 ボードを立てる」系のUIを追加。押すと種類（共有ボード／テキストボード）を選び、**自分の現在位置にボードを設置**。
- ボードは3D内のオブジェクト（板＋脚、種類が分かる見た目。共有ボードは📺/白板、テキストボードは📋/コルク板など）として表示。
- ボード情報（id, 種類, 位置, 立てた人）をデータチャネル（例：topic `boards`）で全員に同期。**接続完了後のみ送信**、自分にはローカル即時反映、後から入室した人向けにハートビート再送（`TentState`のパターン踏襲）。ボードの状態も`TentState`と同様の常時マウントの共有ストア（例：`BoardState.jsx`）で保持し、パネル開閉やビュー再マウントで消えないようにする。
- 立てたボードは動かせなくてよい（まずは設置のみ）。撤去（消す）ボタンはあってもよい（任意）。
- **完了条件**：2タブで、片方が立てたボードがもう片方の世界にも同じ位置に現れる。

### Step 3：共有ボードで画面共有＋描き込み
- 共有ボードを**タップ／接近すると、既存の `ScreenShareStage` + `DrawingOverlay` をフォーカス表示（オーバーレイ）**として開く。閉じると世界に戻る。既存の画面共有・描き込みロジックはそのまま再利用する（作り直さない）。
- 画面共有の開始（`TrackToggle` ScreenShare）は既存のまま。**まずは「共有ボードを開くと、今ルームで共有されている画面が見える」で可**（1ルーム＝1共有の前提でよい）。
- 3Dのボード面には「共有中」状態や小さなプレビューを出すと没入感が上がる（余力があれば：LiveKitの画面共有トラック→`<video>`→`THREE.VideoTexture`をボード面に貼る。難しければアイコン表示でよい）。
- **完了条件**：共有ボードを介して、2タブで片方の画面共有がもう片方に見え、描き込みが同期する。

### Step 4：テキストボードでチャット
- テキストボードを**タップ／接近すると、既存の `ChatPanel` をフォーカス表示**として開く。閉じると世界に戻る。`ChatState.jsx`（常時マウント）はそのまま再利用するので、開閉しても履歴が保持され、閉じている間の受信も取りこぼさない現状の挙動を維持する。
- 3Dのボード面に最新メッセージや件数のヒントを出せると良い（余力があれば。難しければアイコン＋未読的な表現でよい）。
- **完了条件**：テキストボードを介して2タブでチャットの送受信ができる。

## 4. 技術メモ

- 依存はインストール済み：`three` / `@react-three/fiber` / `@react-three/drei`。
- 既存の `client/src/mockup/`（`Mockup.jsx`）は**流用元**。本番`TentView`へ移植したら、mockupページは残してもよいし整理して消してもよい（判断はユーザーに確認）。`mockup.html` と `vite.config.js` のマルチエントリはmockupを消すなら合わせて片付ける。
- データチャネルのメッセージは `TextEncoder`/`TextDecoder`＋JSON（既存踏襲）。topicは機能ごとに分ける（`position` / `chat` / `draw` / 新規`boards`）。
- デプロイは既存のGitHub Actions（`main`へpush→`client`ビルド→Pages）。バックエンド（Render）は今回変更不要。
- ビルドは `cd client && npm run build`、lintは `npx oxlint src/`（既知の`only-export-components`/`exhaustive-deps`警告は許容）。

## 6. 今回やらないこと（スコープ外）
- **永続化・DB・ログ収集**（掲示板テキストやログを“残す”こと）。サーバーを研究室サーバーに移す時に、サーバー＋DB（SPEC 7.4 / F10）でまとめて対応する。今回は全部その場限り（データチャネル）でよい。
- **近接スコープ**（ボード/チャット/画面を「近くの人にだけ」に限定すること）。今回はルーム全体共有でよい。将来の課題。
- 複数テント・通り画面・テーブル自動分割（SPEC Phase 2以降）。
- サーバー変更、認証、AI機能。

## 7. 参考：この構想の位置づけ（背景）
SPECの思想は「Zoom的な全体共有ではなく、空間的・近接的な居場所」。今回はその第一歩として
「2.5Dの世界に集まり、ホワイトボード（共有ボード）と掲示板（テキストボード）を囲んでやりとりする」体験を作る。
近接スコープや永続化（＝研究ログ）は後続で、研究室サーバー移行とセットで入れる。

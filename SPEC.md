# 『通りとテント』アプリ仕様書

デジタル・サードプレイスによるピア・ケア実証のためのウェブアプリ仕様書。
企画書「デジタル・サードプレイスが紡ぐピア・ケアの実証」（長谷川研・JST社会技術研究開発事業提案）に基づく。

- 本書は実装者（AI含む）への指示書として書かれている。実装判断に迷ったら「2. 設計原則」に立ち返ること
- 既存プロトタイプ（本リポジトリ）を土台に**フェーズを区切って少しずつ**拡張する。各フェーズの完了ごとに動作確認してから次に進む
- 対象: 初年度のα版（研究室内クローズドテスト）〜2年度のキャンパス実証を見据えた設計

---

## 1. プロジェクト概要

### 1.1 目的

孤立やメンタルヘルス悪化を抱える層に、専門的ケアの手前にある「ただ他者と共にある」広義のピア・ケアの場をスマートフォン上に提供する。メタバース研究で確認された回復プロセス

**Sanctuary（聖域）→ Sandbox（練習環境）→ Bridge（現実への橋渡し）**

をスマホの空間的UI『通りとテント』で再現する。

### 1.2 実現したい体験（ユーザーストーリー）

1. **聖域として**: 傷ついた学生が「石」のアバターで、声を発さずテントの隅に「ただ居る」。周囲の温かい会話が空間オーディオで聞こえてくる。誰にも発話を求められない
2. **練習環境として**: 石で居続けた学生の心が動き、自発的に「人型」アバターに変わり、小さな会話の輪（テーブル）で声を出してみる
3. **橋渡しとして**: テント間をふらつく「観光客（ホッパー）」が会話の輪に偶然巻き込まれ（誤配）、対面の居場所や専門家の支援につながる

### 1.3 既存SNSとの違い（何を作らないか）

- **フォロー・フォロワー、いいね、投稿の数値評価は作らない**（スケールフリー性・注意経済の排除）
- **1人が大人数に配信する構造は作らない**（Clubhouse型の1対多を避け、3人前後の会話を保護）
- **無限に滞在させる仕組みは作らない**（深夜0時〜朝6時はシステム停止）
- **人間を装うAIは作らない**（AIは明示的なシステムガイド・裏方に徹する）

## 2. 設計原則（実装判断の基準）

| # | 原則 | 実装への含意 |
|---|------|-------------|
| P1 | 受動的な滞在を肯定する | 石アバターで無言のまま居ることを「無価値」として扱うUI（発話促し通知、アクティブ度表示等）を作らない |
| P2 | 会話はフラットに、輪は小さく | 大人数化したら輪を分割する。演者と観客に分かれる構造を防ぐ |
| P3 | 身体の制約をあえて持ち込む | 「近づかないと聞こえない」「移動しないと出会えない」を守る。効率化のためのショートカット（全員に一斉に話す機能等）を安易に足さない |
| P4 | AIは環境であって他者ではない | AIの発言は常にシステムであることを明示。人間のふりをする応答は禁止 |
| P5 | 監視でなく自治 | 問題行動への対応はまず環境フィードバック（Nudge）。BAN等の強権的処置は最終手段 |
| P6 | 休息の境界線 | 深夜停止は「機能」であり、緩和オプションを作らない |
| P7 | モバイルファースト | スマホブラウザで完結する。アプリストア配布・VR機器は前提にしない |

## 3. 用語定義

| 用語 | 定義 |
|------|------|
| **通り（Street）** | アプリのトップ階層。テントが点在するマップ。テントの大きさ（人数）が可視化され、賑わいが環境音として漏れ聞こえる |
| **テント（Tent）** | 会話空間の単位。中に複数のテーブルを持つ。参加人数に応じて見た目が拡大する。技術的にはLiveKitの1ルームに対応 |
| **公共圏テント** | アプリ起動後に最初に表示されるデフォルトのテント。ホッパーが交流する広場 |
| **クローズドテント** | サークル・ゼミ等が作れる非公開テント（招待制） |
| **テーブル（会話の輪）** | テント内の会話グループ。空間上の位置の集まりとして表現される。空間オーディオの焦点単位 |
| **公式テーブル** | カウンセラー等の専門家が滞在するテーブル。視覚的に区別される |
| **アバター** | ユーザーの空間内での姿。「石・草」（話さない意思表示）と「人型」（話す可能性あり）の2系統 |
| **ホッパー / 観光客** | テント・テーブル間を回遊するユーザー。コミュニティのタコツボ化を防ぐ存在 |
| **誤配** | 目的なく回遊するうちに予期しない出会い・会話が生まれること |
| **おやすみ儀式** | 深夜0時の停止直前に行われる「おやすみなさい」の声かけ→テントを畳む演出 |

## 4. 対象ユーザー・利用環境

- **主対象**: 大学生（初年度は長谷川研メンバーとその友人によるクローズドテスト、2年度からキャンパス全体）
- **端末**: スマートフォンのブラウザが主。iOS Safari / Android Chrome の最新2メジャーバージョンで動作すること。PCブラウザ（Chrome/Edge/Firefox）でも利用可能にする
- **前提**: マイク使用許可。イヤホン推奨（空間オーディオの効果とハウリング防止のため。イヤホン推奨の案内をUIに出す）
- **同時接続規模**: α版 = 1テント10名程度 / 本格導入 = 数千ユーザー・同時接続数百を想定した設計（実装はα版規模から始め、スケールは後述のインフラ移行で対応）

## 5. 画面構成

```
[入室/オンボーディング]
   └→ [通り] ←→ [テント内] ←→ (テント内での移動: テーブル間)
                    └→ [チャットパネル] (オーバーレイ)
   └→ [アバター選択] (入室時と、テント内でいつでも変更可)
   └→ [設定] (表示名、音量、通知等)
```

### 5.1 入室・オンボーディング画面

- 表示名の入力（α版は自己申告。本格導入時は大学認証と接続 → F11）
- アバターの初期選択（石 / 草 / 人型数種）。**デフォルト選択は非人間アバター（石）**とし、「まずは気配だけ感じる」参加を最初の選択肢として提示する
- マイク許可の取得はテント入室時まで遅延させる（石で居るだけならマイク不要）
- 初回のみ: アプリの趣旨・ログ収集への同意（研究倫理対応、F10）・深夜停止の説明

### 5.2 通り画面

- テントがアイコンとして点在する2Dマップ（スクロール可能な縦長 or 横長ビュー。スマホの縦持ちを基準にレイアウト）
- 各テントは**参加人数に応じて大きく**描画され、テーマ（テント名）が表示される
- 賑わいの可聴化: 通りを歩く（スクロール・タップ移動する）と、近くのテントから中の会話の環境音（音量を絞り、個別の発言が聞き取れない程度にぼかしたもの）が漏れ聞こえる
- テントをタップ → 入口プレビュー（テント名、人数、公式テーブル有無）→ 入室
- 通りに居るだけでも他ユーザーの気配（通りを歩く他のアバター）が見える

### 5.3 テント内画面

- 2D俯瞰ビュー。テーブル（会話の輪）が複数配置され、各テーブルの周りにアバターが並ぶ
- 自分のアバターをドラッグ or タップ先指定で移動。テーブルに近づくとその輪に「着席」扱い
- テーブルに属さない空間（テントの隅・壁際)にも滞在できる。**隅に居ることは正当な利用**（P1）
- 発話中のアバターは視覚的に示す（波紋・光等の控えめな表現。音量メーター等の競争的表現は避ける）
- 画面下部に操作バー: マイクON/OFF、アバター変更、チャット開閉、周囲の声の遮断、退出（通りへ戻る）
- 「自分の輪以外の方向を向く」操作: アバターの向きを変えると、向いた方向の音が相対的に強調される（F2）

### 5.4 チャットパネル

- テント単位のテキストチャット（既存実装の `<Chat />` を土台に日本語化）
- 声を出せない段階のユーザーの参加手段としても位置づける（石アバターでもチャットは可能）

## 6. 機能仕様

実装フェーズ（§9）との対応を各機能に付記する。

### F1. アバター（Phase 1）

| 項目 | 仕様 |
|------|------|
| 種類 | 非人間系: 石・草（最低2種）。人型: 数種（性別・属性を強く規定しない中立的デザイン） |
| 意味づけ | 非人間 = 「話しません」の明示的シグナル。人型 = 「話す可能性あり」。**この区別が他ユーザーに一目で伝わること**が最重要要件（企画書の初年度検証項目） |
| マイク | 非人間アバター選択中はマイクを強制ミュート（UIでもマイクボタンを非活性に）。人型に変えると発話可能になる |
| 変更 | テント内でいつでも変更可能。変更時に軽い演出（石が人になる変化が周囲にも見える）を入れる。この変更イベントは重要ログ（F10の「石から人への移行」） |
| 表示 | 2Dスプライトで十分（3D不要）。位置・向き・発話中状態を同期 |
| 素材（α版） | **仮デザインでよい**（決定済み）。テキストラベル付きの簡単な図形（例: 灰色の丸＝石、緑の輪郭＝草、輪郭線の人型）で実装を進め、正式デザインは後で差し替える。差し替えやすいよう画像/描画は1モジュールに分離しておく |

### F2. 空間オーディオ（Phase 1 — 本アプリの中核）

- テント内の**アバター間の距離と向きに応じて他者の音声の音量・定位が変化**する
  - 距離減衰: 同じテーブル ≈ 明瞭、隣のテーブル ≈ ざわめき、遠く ≈ ほぼ聞こえない
  - 左右定位: 画面上の相対位置からステレオパンを計算（Web Audio API の `PannerNode` または `StereoPannerNode`）
  - 向き: 自分の向いている方向の音源を相対的に強調（カクテルパーティー効果の補助）
- 実装方式: LiveKitで各参加者の音声トラックを購読し、再生を `WebAudio` ノード経由にして参加者ごとに音量・パンを制御する（LiveKit公式のspatial audioサンプル実装が参考になる）
- 遠方の参加者のトラックは購読自体を停止して帯域を節約（LiveKitの selective subscription）
- 「周囲の声の遮断」ボタン: 自分のテーブル以外の音声をゼロにするアクセシビリティ機能
- **検証条件（α版）**: 8名が1テントに居るとき、テーブルの会話に集中しつつ他テーブルのざわめきが聞き分けられること

### F3. テーブル＝会話の輪（Phase 2）

- テント内に複数テーブルを配置。ユーザーの位置がテーブルの半径内に入るとそのテーブル所属になる
- **自然分割**: 1テーブルの**発話可能者（人型）**が閾値（初期値: 6名）を超えたら、分割を促す
  - **石・草アバターは分割のカウントに含めない**（企画書明記のアルゴリズム要件）
  - α版の分割方法: システムがテーブルを2つに割る視覚演出＋自動で近い者同士を2群に再配置。会話分析ベースの高度な分割（誰と誰が話しているか）はPhase 4以降
- テーブル間の移動は自由。移動はワンタップ（移動先テーブルをタップ→自分のアバターが歩く）
- テーブル数に応じてテントの描画サイズが拡大する（通り画面での見えにも反映）

### F4. テント（Phase 2）

- テント = LiveKitルーム1つ。バックエンドがテントのレジストリ（ID、名称、公開/非公開、作成者、現在人数）を管理
- 公共圏テント: 常設・最初に入る場所。運営（研究チーム）が管理
- クローズドテント（Phase 3）: ユーザー（サークル・ゼミ）が作成可能。招待リンク or 合言葉で入室制限。日常連絡に使える実用性がピア・インビテーション（口コミ普及）の鍵
- テントの定員は設けないが、テーブル分割とテント巨大化で自然に空間が広がる設計とする

### F5. 通り（Phase 2）

- バックエンドの `GET /api/tents` がテント一覧（人数含む）を返し、通り画面が定期ポーリング or WebSocket購読で更新
- 賑わい環境音: 各テントの音声をそのまま流すのは盗み聞きになるため、**人数・発話量から合成した抽象的なざわめき音**（プライバシー配慮）を距離減衰付きで再生する
- 通りに滞在中の他ユーザーのアバターも表示（位置同期は粗い頻度でよい）

### F6. テキストチャット（Phase 1で日本語化、Phase 3でテーブル単位化を検討）

- 既存のLiveKit標準チャットを継続利用。UIラベルを日本語化
- スコープはテント単位から開始。テーブル単位チャットの要否はα版のフィードバックで判断

### F7. 深夜停止・おやすみ儀式（Phase 3）

- **毎日 0:00〜6:00 JST はシステム完全停止**（P6。緩和オプションなし）
- 23:45頃から「テントを畳む」予告演出。0:00に全員を切断し、「おやすみなさい」の声かけを促すUI（定型スタンプ等）を23:45〜0:00に表示
- 停止中のアクセスには静かな夜の通りの画面と再開時刻を表示
- 実装: バックエンドで時刻判定し、トークン発行拒否＋既存ルームの全参加者切断（LiveKit server API）。クライアント側でもカウントダウン表示

### F8. AI機能（Phase 4 — ELSI対応を伴うため後期）

すべて **P4（人間を装わない）** を厳守。AIの発話・介入には必ずシステム表示（例: 🤖ラベル、専用の見た目）を付ける。

1. **明示的システムガイド**: 過疎時間帯・専門家不在時に、新規入室者への使い方案内、公式相談窓口の案内を行うLLMエージェント。常に「AIガイドです」と名乗る
2. **環境的Nudge**: 会話メタ情報（発話時間の偏り等）をテントの色調・照明の変化等の**環境表現**でフィードバックする。特定個人を名指しで注意する通知は出さない
3. **リスク検知フェイルセーフ**: 会話解析で重大リスク（希死念慮等）を検知した場合:
   - 対象者にのみ見えるプライベートな声がけ（AIであることを明示）＋外部24時間相談窓口の案内（一次対応）
   - 記録を専門家に安全に共有し、翌勤務日に大学公式ルートからヒューマン・フォローアップ
   - ※**当面は実装しない**（決定済み）。音声の常時解析はプライバシー影響が大きく、実装範囲・同意取得・データ保持は研究倫理審査と連動して将来別途詳細設計する。それまでこの項目に着手しないこと

### F9. 公式テーブル（Phase 4）

- カウンセラー等が滞在するテーブルを視覚的に区別して常設（公式マーク）
- 専門家アカウント（ロール）を導入し、勤務時間帯のみ「在席」表示
- 専門家が公式テーブルを離れて一般のテーブルに混ざれることも要件（非対称性の解体）。その際は公式マークの表示を抑える等の運用配慮をUIで支援

### F10. ログ収集・解析基盤（Phase 2から段階的に）

研究の効果測定に必須。**取得は本人同意（オンボーディング時）を前提**とし、音声の生データは保存しない。

| ログ種別 | 内容 | 用途 |
|---------|------|------|
| 移動ログ | テント入退室、テーブル間移動、通り滞在 | ホッパー検出・ブリッジング指数 |
| アバターログ | 種類変更イベント（石→人 等） | 「石から人への移行率」 |
| 会話メタログ | 発話時間・発話者交代（内容は保存しない） | 会話の輪の生成・分割の解析、Nudge |
| チャットログ | テキスト（保持期間・匿名化は倫理審査に従う） | 質的分析 |

- 形式: バックエンドで構造化イベント（JSON）として時系列DBまたは追記型ストアに記録。エクスポートしてネットワーク解析（研究側のPython等）に渡せること
- ユーザーIDは仮名化IDで記録し、実名との対応表は分離管理

### F11. 認証・アカウント（α版: Phase 1 / 本格: Phase 5）

- **α版**: 表示名自己申告＋発行済みの合言葉 or 招待リンクでのクローズド運用（研究室内テスト用）
- **本格導入**: 大学認証システム（学認/Shibboleth 等、大学のSSO）との連携。卒業後もアカウントを残す要件があるため、大学IDと本サービスIDは分離して設計しておく
- アカウントに紐づく永続データは最小限（表示名、選択アバター、所属クローズドテント）。プロフィール・実績等の自己呈示機能は作らない（P1）

## 7. 技術アーキテクチャ

### 7.1 方針

- **既存プロトタイプ（本リポジトリ）を土台に拡張する**。React + Vite + LiveKit の構成は本仕様の要件（SFU音声、データチャネル、スマホブラウザ対応）に合致しており継続する
- LiveKitはOSSでセルフホスト可能なため、将来の研究室サーバー移行とも整合する（§7.4）

### 7.2 構成

```
┌────────────────┐     ┌─────────────────────┐     ┌──────────────────┐
│ client (React/Vite) │     │ server (Node/Express)  │     │ LiveKit (SFU)       │
│ 通り/テント/アバターUI │──▶│ ・トークン発行           │──▶│ 音声リレー           │
│ WebAudioで空間音響   │     │ ・テントレジストリAPI     │     │ ルーム=テント        │
│                    │◀──│ ・深夜停止制御           │     │ データチャネル=       │
│                    │     │ ・イベントログ収集        │     │  位置/アバター同期     │
└────────────────┘     │ ・(Phase4) AIガイド     │     └──────────────────┘
                            └─────────────────────┘
                                      │
                              ┌───────────┐
                              │ DB (SQLite→   │
                              │ PostgreSQL)   │
                              │ テント/ログ     │
                              └───────────┘
```

### 7.3 主要な技術判断

| 課題 | 採用方式 | 理由 |
|------|---------|------|
| テントの実体 | LiveKitルーム1つ = テント1つ | 音声のスコープとルームが一致。ルーム一覧・参加者数はLiveKit Server APIで取得可能 |
| 位置・アバター同期 | LiveKitデータチャネル＋participant metadata | 既にDrawingOverlayで実績あり。別途WebSocketサーバーを立てずに済む。位置は0〜1正規化座標、10Hz程度のロスあり配信、アバター変更等の状態はロスなし配信 |
| 空間オーディオ | クライアント側 Web Audio API（トラックごとに GainNode + StereoPannerNode） | サーバー側ミキシング不要でスケールする。LiveKit公式サンプルあり |
| テーブル所属判定 | クライアントで位置から判定し、確定イベントをサーバーに記録 | リアルタイム性優先。ログの正はサーバー記録 |
| 通りの人数表示 | バックエンドがLiveKit Server APIを定期取得しキャッシュ、`GET /api/tents` で配信 | クライアントからLiveKit APIを直接叩かせない（認証情報保護） |
| DB | α版はSQLite、本格導入でPostgreSQL | α版の運用を軽くする。ORMを挟み移行容易に（例: Prisma / Drizzle） |
| AI(LLM) | バックエンド経由でLLM API（Claude API等）を呼ぶ。APIキーはサーバーのみ保持 | Phase 4。クライアントに鍵を置かない |

### 7.4 インフラ（現状 → 将来）

| コンポーネント | 現状（α版） | 将来（研究室サーバー移行後） |
|--------------|------------|--------------------------|
| フロントエンド | GitHub Pages（GitHub Actionsで自動デプロイ） | 研究室サーバーのnginx等で配信（GitHub Pages併用も可） |
| バックエンド | Render 無料プラン | 研究室サーバー上のDocker（docker-compose） |
| SFU | LiveKit Cloud（東京リージョン） | 研究室サーバーでLiveKit OSSをセルフホスト（TURN含む） |
| DB | （新設）Renderのディスク or SQLiteファイル | 同サーバーのPostgreSQL |

移行を見据えた実装ルール:
- 接続先URL・鍵はすべて環境変数化（既存の `.env` 方式を踏襲）
- LiveKit固有APIへの依存箇所は薄いラッパーモジュールに集約しておく
- **Render無料プランのスリープ問題**（初回アクセスで数十秒待ち）はα版では許容。本格実証までに移行

### 7.5 セキュリティ

- CORSは許可オリジンを明示指定に変更する（現状の無制限を是正。Phase 1で対応）
- トークン発行APIに最低限の乱用対策（レート制限、深夜停止時の発行拒否、テントごとのgrant）
- LiveKit API Secret・LLM APIキーはサーバー環境変数のみ。クライアントに露出させない
- クローズドテントの招待リンクは推測困難なランダムトークン

## 8. 非機能要件

- **性能**: 1テント20名・4テーブル程度でスマホ（ミドルレンジAndroid含む）で滑らかに動作すること。位置同期のデータチャネル帯域は1ユーザーあたり数kbps以内に抑える
- **音声品質**: 遅延より安定性優先。遠方トラックの購読停止で帯域を節約
- **アクセシビリティ**: 音声が聞こえない/出せない環境の利用者もチャット＋石アバターで参加可能であること。文字サイズはOS設定に追随
- **多言語**: 日本語のみ（α版）。文言はリソースファイルに分離し将来の英語化に備える
- **プライバシー/ELSI**: 音声の録音・保存はしない。ログは仮名化・同意ベース（F10）。無断配信・録音を禁止する利用規約と通報導線（Phase 3で最低限）
- **可用性**: 研究プロジェクトとして計画停止（深夜停止）以外の稼働率目標は緩やかでよいが、障害時に状態がわかるステータス表示を用意
